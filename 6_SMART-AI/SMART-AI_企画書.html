1.企画概要

本企画は、SMART-AIのPDF・画像PDFのAI読取機能を拡充し、顧客が保有する請求書・納品書・発注書などの帳票データを、SMART-AIで活用可能な構造化データへ変換する機能を、有償の「AIチケット」モデルとともに提供するものである。

既存機能マージ前提。現行のSMART-AIには、DXONE Vision AI によるPDF・画像(JPG/PNG)・Excel・CSVの読取、名寄せ(完全一致/表記ゆれ吸収/別名辞書/外来語・略称/AI判定/新規/非商品)、要マスタ登録の一括登録が既に実装・稼働している。本企画はこれらを新規に作り直すものではなく、既存機能を土台として、不足している要素(名寄せルールの永続化・AIチケット課金・Stripe決済・管理KPI等)を追加(マージ)するものである。既存のVision AI読取・既存名寄せと重複する実装は行わない。

青果卸・食品流通の現場では、取引先から送られてくるデータが必ずしもExcel形式で整備されているわけではない。PDF、スキャンPDF、画像PDF、FAX由来の帳票、取引先独自フォーマットの納品書・発注書が多く存在する。Excelデータを持っている顧客だけでなく、PDFや紙帳票中心の顧客でもSMART-AIを利用できるようにすることが重要である。

本機能では、顧客がPDF・画像PDFをアップロードすると、AIが帳票内容を読み取り、SMART-AI標準データ形式に変換する。変換後は、Excelとしてダウンロードできるだけでなく、SMART-AIに直接取り込むことも可能にする。さらに、青果業界特有の商品名・取引先名・規格・容量・単位・荷姿の表記ゆれに対応するため、名寄せ・読み替え機能を中核機能として設計する。

単なるAI-OCR機能ではなく、読み取ったデータを顧客企業ごとの商品マスタ・取引先マスタに統合し、経営データとして使える状態にすることを目的とする。

2.企画の目的

第一に、SMART-AIの利用対象を拡大することである。データ整備が進んでいない企業(紙・PDF・FAX・画像帳票中心)でもSMART-AIを利用できるようにする。

第二に、PDF・画像PDFを経営データに変換することである。帳票を単に文字起こしするだけでは価値が弱い。商品別・取引先別・日別・粗利別に分析可能なデータへ変換し、SMART-AIの経営分析機能につなげる。

第三に、青果業界特有の表記ゆれを吸収することである。同じ商品でも取引先ごとに呼称が異なる(キュウリ／きゅうり／胡瓜／ｷｭｳﾘ／キューリ／cucumber／胡瓜100g／胡瓜500g／胡瓜L／胡瓜 5kg 等)。これらをそのまま別商品として登録すると、粗利分析・在庫分析・販売分析が壊れる。AI読取後に自社商品マスタへ統合する名寄せ機能が不可欠である。

第四に、SMART ORDER / SMART STOCK / Z-CHAINへつながる正しい取引データを蓄積することである。取引先から届く非構造データを、業界標準化された取引データへ変換する入口であり、将来的にZ-CHAINにおける商品マスタ・取引先データ・受発注データ・納品データ・請求データ・粗利データの蓄積基盤となる。

3.サービス名と購入対象

サービス名は引き続き「SMART-AI」とする。新しい別サービス名は作らない。

今回追加する有償購入対象は「AIチケット」とする。AIチケットとは、PDF・画像PDFをAIで読み取り、SMART-AIで使えるデータへ変換するための利用チケットである。

顧客向け画面では以下の表現に統一する:
AIチケット — PDF・画像PDFをAIでデータ化するためのチケットです。

顧客向けには、Claude、トークン、API原価、AIクレジットという表現は使用しない。内部的にはAI APIを利用するが、顧客にとって重要なのはAIの仕組みではなく、PDFをデータ化できることである。

4.AIチケットの料金設計

項目内容
1回あたりの価格換算250円相当
1回あたりのページ上限5ページまで
有料チケット20回分 10,000円
20回で変換できる最大ページ数100ページ
初回無料枠3回分

チケット消費条件

  • 1回のAI変換につき、AIチケットを1回分消費する。
  • 変換に成功した場合のみ消費する。変換失敗時は消費しない。
  • 同一ファイルの再変換が無制限になると原価リスクがあるため、再実行制限を設ける。
ページ上限チェックは「1回5ページまで」で実装する。Stripe側に作成済みの商品(¥10,000)をそのまま利用する。

4-2.AIチケットのSO/SS共有

要件

  • AIチケットは、SO(スマートオーダー)側SMART-AIとSS(スマートストック)側SMART-AIで共有とする。
  • 同一のユーザー/テナント(会社)であれば、チケット残高は1つであり、SO側で使ってもSS側で使っても同じ残高から消費される。
  • 購入(Stripe決済)も同様に、どちらの画面から購入しても付与先は同じ1つの残高である。
  • 例:残20回の状態でSO側で1回使えば、SS側で見ても残19回。SS側で購入すればSO側でも残高が増える。

設計原則(財布を2つ作らない)

  • チケット残高・付与/消費履歴は、SOとSSで別々に持たず、単一の保存場所(single source of truth)とする。
  • 実現方式は実装specで比較・確定する:①共通DB(共通スキーマ)に置き両システムから参照・更新/②一方をマスタとし他方はAPI経由で照会・消費/③現行のSO/SS構成(認証・テナント対応関係)に照らした最適案。
  • SO/SS双方から同時に消費が走っても、残高マイナス・二重消費が発生しないこと(排他/トランザクション整合)。
  • 消費・付与の履歴に「どちらのシステムで使ったか(SO/SS)」を記録し、管理画面で確認できること。
  • 残高表示は両システムで一致すること(片側で使った直後に他方でも正しい残数が見える)。
  • SOテナントとSSテナントの同一性の突き合わせ方法(同一ID/別採番/既存連携の利用)は、実装前調査で確定する。

決済後の処理フロー

  1. ユーザーがAIチケット購入ボタンを押す
  2. Stripeの決済画面へ遷移する
  3. ユーザーがStripe画面で決済する

  6. 購入履歴に記録する(stripe_payment_id/イベントIDを保存)

【設計上の最重要論点】購入者⇔テナントの紐づけ。作成済みのPayment Linkは全顧客共通のため、そのままでは「誰が買ったか」をテナントへ自動で紐づけられない。実装specで以下を比較し方式を確定する:
案1:Payment Linkに client_reference_id(テナントID)等のパラメータを付与してアプリから遷移させ、Webhookで受け取り紐づける。
案2:アプリ側でStripe Checkout Sessionを都度作成する(既存商品 prod_UsA5lGu2pJV0o8 の価格を参照)。テナント紐づけが最も確実。
いずれの場合も既存商品を活かし、Stripe側に重複商品を作らない。Webhook再送・二重通知でも二重付与しない(イベントIDで冪等化)。決済失敗時は付与しない。APIキー・Webhookシークレットはコードに直書きせず環境変数等で管理する。

6.顧客向け体験フロー

  1. SMART-AIトップ画面にアクセスする
  2. 残りAIチケット数を確認する
  3. PDFをAIで変換するボタンを押す
  4. PDFまたは画像PDFをアップロードする
  5. 書類種別を選択する
  6. 必要に応じて画像解析モードをONにする
  7. AIが帳票を読み取る
  8. SMART-AI標準データ形式に変換される
  9. 名寄せ候補が表示される
  10. ユーザーが必要に応じて修正する
  11. 名寄せルールを保存する
  12. Excelでダウンロードする/またはSMART-AIへ直接取り込む
  13. 変換成功時にAIチケットが1回分消費される

顧客にとっての価値:紙・PDFの帳票を手入力しなくてよい。取引先ごとに違う商品名を自社マスタへ統合できる。Excelデータがない会社でもSMART-AIを使える。使えば使うほど名寄せルールが蓄積され、次回以降の作業が楽になる。SMART ORDER / SMART STOCKへ正しい商品マスタIDでつながる。

7.対象帳票

MVPでは、対象帳票を請求書中心に絞る。対応候補:請求書/納品書/発注書/発注依頼書/仕入明細/販売明細。初期リリースでは請求書を優先し、後続フェーズで納品書・発注書・発注依頼書に拡張する。

初期対象外:手書き帳票/極端に画質が悪いFAX/複数帳票が混在したPDF/上限ページ数を大きく超えるPDF/罫線が大きく崩れた帳票/明細行の構造が判別できない帳票。

8.PDF読取機能の基本仕様

PDF・画像のAI読取は既存のDXONE Vision AIで既に稼働している。本章の要件のうち既存で充足するものは既存を流用し、不足分(ページ上限チェック・アップロード時チェックの拡充・画像解析モードの明示切替等)のみ追加する。

処理対象:テキストPDF/スキャンPDF/画像PDF/JPG/PNG。PDFがテキストを含む場合はテキスト抽出とAI解析を組み合わせ、画像の場合は画像解析モードでAIに読み取らせる。

アップロード時のチェック:ファイル形式/ファイルサイズ/ページ数/重複ファイル/破損ファイル/パスワード付きPDF/読取不能PDF。

1回あたりのページ上限は5ページとする。

9.画像解析モード

スキャンPDFや画像PDFでは通常のテキスト抽出だけでは読み取れないため、PDFを画像としてAIに読み取らせる「画像解析モード」を設ける。顧客向け表示:「画像PDF・スキャンPDFの場合は、画像解析モードをONにしてください。」

画像解析モードはAI API原価が高くなる可能性があるため、管理画面では通常PDFと画像解析モードの利用状況を分けて確認できるようにする。初期リリースでは画像解析モードもAIチケット1回分で処理する。原価が高すぎる場合は、将来的に高精度モードとして2チケット消費も検討する。

10.SMART-AI標準データ形式

PDFから読み取った内容は、まずSMART-AI標準JSONに変換し、その後、名寄せ処理・Excelエクスポート・SMART-AI直接取込に流す。

標準JSONの項目:document_type / supplier_name / customer_name / document_date / document_number / subtotal / tax / total / line_items / warnings

line_itemsの項目:raw_product_name / base_product_name / normalized_product_name / product_master_id / quantity / unit / normalized_unit / unit_price / amount / origin / normalized_origin / grade / normalized_grade / capacity / capacity_unit / package_type / confidence / matching_status

matching_statusの区分:matched / candidate / unmatched / manually_mapped / new_master_required

11.名寄せ機能の重要性

本企画の中核は、PDFを読むことではない。本質は、PDFから読み取ったデータを、自社の経営データとして使える状態に変換することである。青果業界では同じ商品でも取引先ごとに呼び方が異なり(キュウリ/きゅうり/胡瓜/ｷｭｳﾘ/CUCU/胡瓜100g/胡瓜500g/きゅうり袋/きゅうりバラ 等)、これらをすべて別商品として扱うとSMART-AI上の分析が壊れる。読み取った商品名を自社商品マスタに統合する名寄せ機能が必須である。

名寄せの照合エンジン(完全一致/表記ゆれ吸収/別名辞書/外来語・略称/AI判定/新規/非商品の経路分類・信頼度表示)は既に稼働している。本企画で追加する中核は、名寄せルールの永続保存・取引先別優先・次回自動適用(§15〜18)=「使うほど精度が上がる」蓄積の仕組みである。

12.青果向け商品名正規化の考え方

名寄せは単なる文字列一致では不十分であり、以下を組み合わせる:文字正規化/商品名分解/類似度判定/過去ルール参照/取引先別呼称参照/自社マスタ候補提示/ユーザー確認/ルール保存/次回自動適用。

文字正規化:全角・半角の統一/カタカナ・ひらがなの統一/英大文字・小文字の統一/不要な空白・記号の除去/長音・表記ゆれ補正/旧字体・異体字補正/英語名・略称の日本語変換。

商品名分解:基本商品名/規格/等級/サイズ/容量/単位/荷姿/産地/ブランド名/顧客独自略称 を分離する。

13.商品名と規格情報の分離

特に重要なのは、商品名と規格情報を混同しないことである。「キュウリ100g」「キュウリ500g」は商品名としてはどちらもキュウリであり、容量が異なるだけである。

読み取り文字列分解結果
キュウリ500graw_name: キュウリ500g / base_product_name: キュウリ / normalized_product_name: 胡瓜 / capacity: 500 / capacity_unit: g / package_type: null / grade: null / product_master_id: 自社商品マスタID
胡瓜L 5kgraw_name: 胡瓜L 5kg / base_product_name: 胡瓜 / normalized_product_name: 胡瓜 / grade: L / capacity: 5 / capacity_unit: kg / product_master_id: 自社商品マスタID

読み取り結果をそのまま商品名に入れるのではなく、商品名・規格・容量・荷姿・単位を可能な限り分解して扱う。

14.名寄せ候補生成

名寄せ候補は以下の情報を使って生成する:顧客の商品マスタ/顧客の取引先マスタ/過去に保存された名寄せルール/取引先ごとの過去の呼称/商品名の類似度/読み方の類似度/表記ゆれ辞書/青果品目辞書/単位・規格・容量の分解結果。

候補には信頼度スコアを付与する。例:AI読み取り名「ｷｭｳﾘ500g」→ 候補1:胡瓜/容量500g/信頼度95%、候補2:きゅうり/容量500g/信頼度88%、候補3:胡瓜L/信頼度62%。信頼度が高い候補は自動候補として表示し、低い候補は要確認とする。初期段階では完全自動確定は避け、プレビュー画面で確認可能にする。

15.名寄せルール管理

SMART-AI上に「名寄せルール管理」(読み替えルール管理)ボタンを配置し、保存済みルールの確認・編集・削除を可能にする。

管理項目:ルール名/対象種別/変換前表記/正規化後表記/変換後マスタ項目/product_master_id/partner_master_id/適用範囲/適用条件/優先順位/有効・無効/使用回数/最終使用日/作成者/更新者。

対象種別:商品名/取引先名/単位/規格/容量/荷姿/産地。適用範囲:顧客全体/取引先別/帳票種別別。ルール種類:完全一致/部分一致/正規表現/同義語/AI提案/取引先別優先/顧客全体。

16.名寄せルールの保存単位

名寄せルールは顧客単位で管理し、他社のルールが別会社に混ざってはならない。必ずcompany_idまたはtenant_id単位で分離する。取引先ごとに呼称が異なる場合に対応するため、partner_id単位のルールも持てるようにする(例:A社からの「CUCU」は胡瓜、B社からの「CUCU」はミニ胡瓜)。取引先別ルールの優先度を顧客全体ルールより高く設定できるようにする。

17.名寄せ画面

変換結果プレビュー画面に名寄せ確認エリアを設ける。表示項目:AI読み取り名/分解された基本商品名・容量・単位・規格・荷姿/自社マスタ候補/選択中の変換先/信頼度/要確認フラグ/「この対応関係を次回以降も使う」チェック/取引先別ルールとして保存するか/顧客全体ルールとして保存するか/新規マスタ登録ボタン。

18.使うほど精度が上がる体験

名寄せは初回から完全自動にしない。初回:AIが読み取り候補を提示し、ユーザーが正しい自社マスタを選択、ルールとして保存する。2回目以降:保存済みルールに基づき自動で自社マスタに変換し、不明なものだけユーザーに確認させる。この流れにより、使えば使うほど名寄せの手間が減り、SMART-AIが顧客企業ごとの業務に馴染んでいく。

19.画面構成

SMART-AIトップ画面:Excel/CSVアップロード/PDFをAIで変換する/残りAIチケット数/無料AIチケット残数/AIチケットを購入するボタン/名寄せルール管理ボタン。

20.処理フロー

  1. ユーザーがPDFをアップロード
  2. ファイル形式・ページ数・容量をチェック

  4. PDFがテキストPDFか画像PDFかを判定
  5. 必要に応じて画像解析モードで処理
  6. AI APIで帳票内容を読み取る
  7. SMART-AI標準JSONへ変換
  8. 商品名・取引先名・単位・規格を分解
  9. 自社商品マスタ・取引先マスタと照合

  13. 名寄せルールを保存
  14. 明細合計と帳票合計をチェック
  15. Excelエクスポート/SMART-AI直接取込
  16. 成功時にAIチケットを消費(共有残高から)
  17. 変換履歴・チケット履歴を保存

26.原価管理・粗利管理

AIチケットは従量課金型の収益機能であるため、売上だけでなくAI API原価を必ず管理する。管理指標:1回あたり平均API原価/1ページあたり平均API原価/画像解析モード時・通常PDF時の平均API原価/変換失敗時・再変換時のAPI原価/AIチケット売上・粗利/顧客別利用回数・粗利。

1回250円相当で販売するため、API原価が高すぎる場合は価格設計を見直す。将来的には、通常PDF:1チケット/画像解析モード:1チケット/高精度画像解析モード:2チケット/5ページ超過:追加チケット/大量処理:別料金 も検討する。

27.セキュリティ・ファイル保存方針

PDF・画像PDFには取引先名・金額・商品名・請求情報などの機密情報が含まれるため、以下を方針とする:ファイルはcompany_id単位で分離し他社ファイルにアクセスできないようにする/アップロードファイルにアクセス権限を設定/不要ファイルは一定期間後に削除/変換結果JSONも会社単位で分離/AI APIに送信するデータ範囲を最小限に/ログに個人情報・機密情報を過剰に残さない/管理者操作ログを残す/Stripe決済情報は自社DBにカード情報として保存しない。

28.例外処理

  • ページ数超過:5ページを超える場合はアップロード不可、または分割を促す。
  • ファイル形式不正:PDF/JPG/PNG以外はエラー。
  • パスワード付きPDF・破損PDF:読取不可としてエラー。
  • AI読取失敗:AIチケットは消費しない。
  • 合計不一致:プレビュー画面で警告し、直接取込を制限する。
  • 未名寄せ多数:確認を促す。
  • Stripe決済失敗:AIチケットは付与しない。
  • Stripe Webhook重複:二重付与を防止する(イベントIDで冪等化)。
  • 同一ファイル再変換:一定回数以上は制限する。

31.実装ステップ

Step 1(現状仕様確認)を必ず最初に行い、既存実装(Vision AI読取・既存名寄せ・一括登録・取込フロー)との突き合わせ(ギャップ分析)を経てから各Stepに着手する。既存で充足する部分は再実装しない。
  1. 現状仕様確認:Excel/CSVアップロード処理/smart_ai_draft・smart_ai_extract関連処理/商品マスタ・取引先マスタ構造/SO・SSとの連携方式/SOテナント⇔SSテナントの対応関係(チケット共有の前提)。
  2. AIチケット基盤:共有残高テーブル(§4-2)/無料3回付与/消費履歴/失敗時非消費/5ページ上限/再変換制限。
  3. Stripe連携:紐づけ方式の確定(§5 案1/案2)/Webhook/決済完了後の付与/二重付与防止。テストモードで通し検証後に本番へ。
  4. PDFアップロード:ファイルチェック/ページ数チェック(5ページ)/変換ジョブ管理。既存アップロードの拡張を優先。
  5. AI読取:既存Vision AIの流用を基本とし、標準JSON変換・信頼度・警告の不足分を補強。
  6. 名寄せ:既存照合エンジンの上に、文字正規化・商品名分解の強化、候補スコアリング、名寄せルール保存(中核)を追加。
  7. プレビュー:要確認フラグ/合計チェック/名寄せ候補選択。
  8. 出力・取込:Excelエクスポート/SMART-AI直接取込/取込前チェック。既存取込フローと整合。
  9. 管理画面:チケット履歴/変換履歴/API原価/名寄せ成功率。
  10. テスト:PDF読取/画像PDF/名寄せ/Stripe決済/チケット消費(SO/SS共有含む)/SMART-AI取込。

32.テスト観点

AIチケット関連

  • 新規顧客に無料3回分が付与されるか。
  • 変換成功時のみチケットが消費され、失敗時は消費されないか。
  • 無料チケットから先に消費されるか。有料チケット購入後に残数が増えるか。
  • 二重決済・二重付与が発生しないか。
  • SO側で購入→SS側でも残高が増えるか(逆も同様)。SO側で消費→SS側の残数が一致して減るか。SO/SSほぼ同時消費でも二重消費・マイナスが起きないか。履歴にSO/SSどちらで使ったかが記録されるか。

Stripe関連

  • 決済画面に遷移するか。決済完了Webhookを受け取れるか。
  • 失敗時に付与されないか。Webhook再送時に二重付与されないか。
  • 購入者が正しいテナントに紐づき、別テナントへ誤付与されないか。

PDF読取関連

  • テキストPDF・画像PDFを読み取れるか。
  • 5ページ超過時に制限されるか。
  • 破損PDF・パスワード付きPDFでエラーになるか。

名寄せ関連

  • キュウリ/きゅうり/胡瓜/ｷｭｳﾘが同一候補になるか。
  • キュウリ100g・キュウリ500gを商品名と容量に分解できるか。
  • 取引先別ルールが優先されるか。顧客別ルールが他社に漏れないか。
  • 保存したルールが次回適用されるか。

SMART-AI取込・管理画面関連

  • 合計一致時に取込でき、不一致時に警告が出るか。未名寄せ商品を確認できるか。商品マスタID付きで取込できるか。
  • チケット残数・変換履歴・API原価・粗利・名寄せ成功率が管理画面で確認できるか。

35.経営上の意義

本機能は、SMART-AIの有償機能として売上を作るだけでなく、SMART-AIの対象市場を広げる意味がある。Excel/CSVデータを持つ企業だけでなく、PDF・紙・FAX中心の企業もSMART-AIに取り込めるようになる。AIチケットにより従量課金モデルを組み込めるため、月額課金に加えて利用量に応じた売上を作ることができる。

また、PDF読取と名寄せによって、顧客ごとの商品名・取引先名・取引データが蓄積される。これは将来的に、SMART ORDER、SMART STOCK、Z-CHAINへ接続するための重要なデータ資産となる。

36.企画の本質

本企画の本質は、PDFを読むことではない。本質は、取引先ごとにバラバラな帳票・商品名・呼称・単位・規格を、顧客企業の自社マスタに統合し、経営データとして使える状態にすることである。

PDF読取だけなら他社AI-OCRでもできる。しかし、青果業界における表記ゆれ・荷姿違い・容量違い・規格違い・取引先別呼称を吸収し、SMART-AI、SMART ORDER、SMART STOCKに正しい商品マスタIDでつなげられることが、SMART-AIの競争優位である。

重要KPI:AI読取成功率/自動名寄せ成功率/未名寄せ・要確認件数/AIチケット利用回数/AIチケット粗利率/SMART-AI直接取込率。

この機能により、SMART-AIは単なるExcel分析ツールではなく、青果卸・食品流通における非構造帳票を経営データへ変換する業務OSへ進化する。
