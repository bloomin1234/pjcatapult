01現状の整理Diagnosis ／ 状況把握

デザインの前に、整理がある。私の方法は、三段階。状況を把握し、視点を導入し、課題を設定する。そのうえで本質をつかみ、それ以外を削る。アウトプットの精度は、思考の整理で決まる。まず、いまの当社を、ありのまま見る。

Step 01
状況把握
いま、何が起きているか。思い込みを捨て、事実だけを並べる。
Step 02
視点導入
どの角度から見るか。会社の本質を言い当てる一つの視点を持つ。
Step 03
課題設定
登るべき山を決める。問題を、解くべき形に整える。

状況把握 ― ほどくべき、三つの重なり

重なり 01RED

赤が、二つの仕事をしている。

赤はブランドマークであり、同時に必須・警告の信号でもある。器が同じだと、ロゴは鳴り続ける警告音になり、本当の警告は埋もれて効かなくなる。これが「謎の赤」の正体。

重なり 025 COLOURS

色が、増えてしまった。

正典は「3色＋無彩色」。だが現場では緑・紺・茶・墨・赤の5色が走っている。意味を持たない色が増えるほど、視線は迷い、画面は遅くなる。

重なり 03NO SPINE

背骨が、見えていない。

Smart Order、Smart Stock、Smart-AI、S-Mobile、Smart-EMS、スマートコネクト。名前は多いのに、それらを束ねる一本の背骨——構想の名——が外に出ていない。

結論。　足すのではなく、減らす。一つの色には一つの意味だけを背負わせ、増えた色を退役させる。そして散らばった製品名に、一本の背骨を通す。リブランドとは、新しさを足すことではなく、本質を取り出すための整理である。
02核心The Core Idea ／ 視点導入・課題設定

視点を、定める。社長が反応するのは機能説明ではなく、「何の会社か」という一文。ロゴとは、伝えたい思いを凝縮したコミュニケーションツール。その凝縮の前に、何を言い切るかを決める。

当社は、受発注システムを売る会社ではない。
当社は、Smart Order／Smart Stock を入口に、青果・食品流通の受発注・在庫・売買データをデータカプセル化し、入力業務と営業活動をAI化することで、中間流通の潤滑機能を高性能化する 「Zチェーン」 を構築する。
仲抜きではなく、潤滑機能の高性能化。

物流・加工・品質確認・需給調整という価値ある機能は残す。消すのは、FAX・LINE・Excel・手入力・転記・照合・集計・無意味な営業資料作成といった低付加価値作業だけ。だから既存業界から敵視されにくい。

入口と、目的地。

Smart Order／Smart Stock は単体では業務システム。だが戦略上はZチェーンを構成する入口プロダクト。1社への導入は売上であると同時に、ネットワークのノードを1つ増やす活動である。

課題設定 ― 問題点を、付加価値に転化する。　全体を壊していた「赤」は、捨てる欠点ではない。赤を信号という一つの仕事に還せば、警告はむしろ強く効くようになる。弱点だった色が、規律の起点に変わる。整理とは、欠点を価値に裏返す作業でもある。

形と面

Form & Surface
  • 直角。角丸は border-radius 2px に統一。曖昧な丸みで親しみを装わない。
  • 平面。影・グラデーションは 0。立体ではなく、面と線で階層をつくる。
  • 線で分ける。境界はヘアライン #E6E6E6、構造は墨 #0A0A0A の太線で締める。
  • 余白とグリッド。間隔は 8px を基本単位。詰め込まない。

タイポグラフィ

Typography
  • 一書体に統一。Noto Sans（日英共通）。Archivo のドリフトは退役。
  • 階層は太さで。大きさ・太さ・余白の差だけで 900 / 700 / 400。
  • 英字は大文字＋字間。補助情報として静かに添える UPPERCASE · tracking。
  • 数字は桁揃え。業務データは誤読を防ぐ。行間 1.7。

運用の手順

Governance
  1. 新規・改修の着手時に、本ポリシーを参照する。
  2. 共通コンポーネント・カラー・タイポを再利用する。
  3. 逸脱が必要な場合は、デザイン責任者のレビューを受ける。
  4. 承認された変更は、本書へ反映し全体へ展開する。

版の管理

Versioning

本ポリシーは生きた文書。四半期ごとに見直し、事例の蓄積に応じて更新する。最新版を常に唯一の基準とし、旧版は参照しない。残すのは、最終バージョンだけ。

08モバイル ＆ UI 設計仕様Mobile & UI Spec

システムはモバイルを前提に設計する。既定はモバイル、デスクトップは768px以上の拡張。指が届く下部に操作を置く。以下は、二度同じ指示をしないための事細かな決定仕様——当社の基本デザイン（モバイル管理）に準拠する。

8.1原則Principles

設計の前提

Base
  • モバイルファースト。既定はモバイル、min-width:768px で拡張。
  • 操作は下に。移動・主アクションは親指が届く画面下部へ。右上に主操作を置かない。
  • セーフエリア。env(safe-area-inset) を確保。タップ最小44〜56px。
  • 機能は不可侵。データと業務ロジックは共通。変えるのは配置だけ。

佐藤可士和の規律（再掲）

Discipline
  • 平面。影・グラデーションは使わない。境界はヘアライン、構造は墨の線。
  • 四角（角丸0）基調。シート・ボタン・行・×・バッジは角丸0。細部のみ最大2px。
  • 一書体。Noto Sans／JP。階層は太さ（900／700／400）。数字は等幅。
  • 色は意味に。主役色は1画面に一つ。状態は色だけに頼らず形・位置・ラベル併用。

8.7状態と、動きStates & Motion

動きは、佐藤可士和の規律の中で。ブランドの動きは LOGO PLAY（第9章）、製品画面の動きはここで定める——意味のある最小限の遷移と微動だけ。明るい緑基調を保ち、派手な演出には振らない。

状態

States
  • 色だけに頼らない。形・位置・ラベルを併用（中空ドット＝白塗り＋墨縁 など）。
  • フォーカスは墨の輪郭線、またはやわらかな緑グロー。
  • disabled は面を #F4F4F4、文字は #9A9A9A。
  • 必須・エラーは赤の左罫／必須バッジ。多用しない。

動き（微動）

  • 金額ポップ。再計算した数字がバネで弾む（overshoot）。

09動き、という層Motion ／ 中村勇吾

静止の形（佐藤可士和）の上に、もう一つの層——動き（中村勇吾）。ユニクロが、佐藤の固定ロゴと中村の LOGO PLAY で一つの世界になったように。中村勇吾は橋梁設計＝構造の出身で、装飾を足さず構造そのものを表現にする（tha の標語 “composition of logics”）。形は描き変えず、決まったロゴを単位として、配置・時間・色だけを規則的に動かす——それが彼の作法である。

シンプルなロゴが、白い盤面で静かに動く。Zマークは変えない。グリッド＝企業のネットワークの上で、ロゴがパタパタとめくれ、緑の波（売る）と紺の波（買う）が渡る——企業から企業へ流れる、二つの取引。にぎやかな演出ではなく、規則的で、静かな反復。

構造から発想
橋梁出身。構造そのものを表現にする。仕組み（ロジック）を組み、走らせる。
形は不変、仕組みが動く
ロゴは描き変えない。動くのは配置・時間・色だけ（UNIQLO LOGO PLAY と同じ作法）。
意味のある動きだけ
装飾はしない。波は、売る／買うの流れそのもの。
二人は対等
形は佐藤、仕組みは中村。静と動が、同じ一点で出会う。
動きの置き場所。　ブランドの動きは、白い地の LOGO PLAY（ZChain_LogoPlay_Nakamura）——固定ロゴが盤面でパタパタとめくれ、緑＝売る／紺＝買うの波が画面いっぱいに渡る、静かな反復。日常の業務画面では、動きは §8.7 の最小限の微動に留める。静止（佐藤）が土台、動き（中村）はその上の層——どちらも対等な頂点として、当社の地＝白の上に置く。

  if(!page || typeof html2canvas==='undefined'){ window.print(); return; }  // 非対応は印刷へ
  const canvas=await html2canvas(page,{
    scale: Math.min(2,(devicePixelRatio||1)+1), backgroundColor:'#ffffff', useCORS:true,
    width:page.scrollWidth, height:page.scrollHeight, scrollX:0, scrollY:0 });
  const imgJpeg=canvas.toDataURL('image/jpeg',0.92);
  try{ _imgBlob=await (await fetch(imgJpeg)).blob(); }catch(e){ _imgBlob=null; }
  showResultImage(imgJpeg);          // できあがり画像を必ず画面に出す＝保存できる
}
// 生成画像を全面表示。下の画像を長押し→「"写真"に追加」で保存できる
function showResultImage(imgUrl){
  const ov=document.getElementById('print-overlay'); if(!ov) return;
  const layer=document.createElement('div'); layer.id='po-result';
  layer.innerHTML=
    '<div class="po-rbar">'
    +'<button onclick="hideResultImage()">← 戻る</button>'
    +'<button onclick="shareResultImage()">画像を保存</button></div>'
    +'<div class="po-rhint"><b>画像で保存：</b>下の表を<b>長押し</b>して'
    +'「"写真"に追加」を選ぶと保存できます（スクリーンショットでもOK）。</div>'
    +'<img class="po-rimg" src="'+imgUrl+'" alt="出力画像">';
  ov.appendChild(layer);
}
function hideResultImage(){ const l=document.getElementById('po-result'); if(l) l.remove(); }
// 共有シートへ（iOS：そのまま「写真に保存」できる）／PCは download にフォールバック
async function shareResultImage(){
  if(!_imgBlob){ alert('画像を生成できませんでした。表を長押しして保存してください。'); return; }
  try{
    const file=new File([_imgBlob], _imgName, {type:'image/jpeg'});
    if(navigator.canShare && navigator.canShare({files:[file]})){
      await navigator.share({files:[file], title:_imgName}); return;
    }
  }catch(e){ if(e && e.name==='AbortError') return; }
  const isIOS=/iPad|iPhone|iPod/.test(navigator.userAgent)
    || (navigator.platform==='MacIntel' && navigator.maxTouchPoints>1);
  if(isIOS){ alert('下の表を「長押し」→「"写真"に追加」で保存できます（スクリーンショットでもOK）。'); return; }
  const url=URL.createObjectURL(_imgBlob);
  const a=document.createElement('a'); a.href=url; a.download=_imgName;
  document.body.appendChild(a); a.click(); a.remove();
  setTimeout(()=>URL.revokeObjectURL(url),10000);
}
